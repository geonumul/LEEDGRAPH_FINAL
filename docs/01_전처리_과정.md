## 2. CSV 매칭 전략 — PDF·CSV 불일치(460 ≠ 456) 처리

### 2.1 매칭 우선순위 (fallback 3단계)

```
1순위: project_id 정확 매칭      → 건물명 철자 달라도 100% 확실
2순위: project_name 소문자 비교   → id 실패 시 이름으로 재시도
3순위: 매칭 실패 → None           → PDF 데이터만으로 project 구성
```

### 2.2 매칭 실패 시 처리

CSV 와 매칭 안 돼도 **분석에서 제외하지 않음**.
- 매칭 성공 시: `certification_level`, `GrossFloorArea` (sq ft → sqm 변환: × 0.0929) 등 메타 보강
- 매칭 실패 시: PDF 파싱값만으로 project dict 구성 (건물유형 = "", 연면적 = 0.0)

### 2.3 460 ≠ 456 원인 (가능성)

- CSV 갱신 시차 (scorecard 만 확보된 건)
- 1 프로젝트가 여러 scorecard 로 분리된 건 (예: IFC 1F + IFC B3, Alphadom 6-1 + 6-2)
- USGBC 에서 confidential 처리된 프로젝트 (CSV 누락)

→ 불일치 건도 PDF 데이터만으로 표준화 가능. 결과적으로 `final_v5_data` 에는 460 행.

## 3. PDF 파싱

### 3.1 추출 항목

`pdfplumber` 로 텍스트 추출 후 정규식 파싱.

| 필드 | 추출 근거 |
|------|----------|
| `version` | `(v4.1)`, `(v2009)` 등 괄호 패턴 |
| `leed_system` | `LEED O+M: ... (v4)` 형태 |
| `certification_level` | `PLATINUM/GOLD/SILVER/CERTIFIED` 키워드 |
| `project_id` | 7자리 이상 숫자로 시작하는 줄 |
| `total_awarded / total_possible` | `TOTAL 64 / 110` 패턴 |
| `categories` | `SUSTAINABLE SITES AWARDED: X / Y` 등 9개 카테고리 |
| `credits` | `^(Prereq|Credit) <이름> X / Y$` 패턴 |

### 3.2 USGBC PDF 의 "이중 문자" 버그 대응

일부 스코어카드에 `SSUUSSTTAAIINNAABBLLEE` 처럼 문자가 2번씩 중복되는 인코딩 버그 존재.
- 대문자 2쌍 이상 반복 → 1글자로 축약 (단, `EE` 1쌍은 보존해 `LEED` 유지)
- 숫자·기호 1쌍 이상 반복 → 1글자로 축약 (`::` → `:`, `//` → `/`)

### 3.3 파싱 결과 예시

```
project_id:          1000072465
project_name:        NAMSAN SQUARE
version:             v2009
leed_system:         LEED BD+C: Core and Shell (v2009)
certification_level: Gold
total_awarded:       61
total_possible:      110

categories:
  SS:  awarded=19, possible=26
  WE:  awarded=5,  possible=10
  EA:  awarded=18, possible=35
  MR:  awarded=5,  possible=14
  IEQ: awarded=10, possible=15
  IN:  awarded=4,  possible=6
  RP:  awarded=4,  possible=4

credits:
  Credit: Alternative Transportation - Public Transit  → awarded=6, possible=6
  Credit: Optimize Energy Performance                   → awarded=10, possible=19
  ...
```

## 4. 버전 표준화 — 비율 환산 공식 (핵심)

### 4.1 수식

```
                  awarded (구버전 획득점수)
score_v5(cat) = ───────────────────────── × v5_max(cat)
                  old_max (구버전 만점)

단, 결과값을 v5_max 로 클램핑(상한 적용)하여 초과 방지.
```

**의미**: "이 카테고리에서 몇 % 를 달성했는가" 라는 **달성률(achievement ratio)** 을 보존한 채 v5 만점 스케일로 재배정. 달성률이 변하지 않으므로 ML feature 로 `ratio_{cat} = score_v5 / v5_max` 를 쓰면 버전 간 동일 기준 비교 가능.

### 4.2 수식이 내뱉는 값

소수점 2자리 **v5 카테고리별 점수** (0 ~ v5_max(cat) 범위).

**예시** — v2.2 의 EA (17점 만점) 에서 12점 받은 건물을 v5 EA (33점 만점) 로 환산:

```
score_v5(EA) = (12 / 17) × 33 = 23.29   → 23.29점 (v5 EA 기준)
```

달성률 70.6% 가 그대로 보존됨 (`12/17 = 23.29/33 = 0.706`).

### 4.3 왜 "비율 환산" 인가?

| 버전 | 총점 | EA 만점 | EA 비중 |
|------|------|---------|---------|
| v2.2 | 69 | 17 | 24.6% |
| v2009 | 110 | 35 | 31.8% |
| v4 | 110 | 33 | 30.0% |
| v5 | 100 | 33 | 33.0% |

버전마다 카테고리 총점·비중 다름. v2.2 EA 17점과 v4 EA 17점은 **의미가 전혀 다름** (전자 100% 달성, 후자 52% 달성). 상대적 달성률 기준으로 환산해야 공정 비교 가능.

### 4.4 버전별 매핑 전략

#### (A) v4.1 / v5 → 직접 매핑

구조 완전 동일. 카테고리 점수 그대로 사용.

#### (B) v4 → WE 만 비율 환산

v4 와 v5 의 유일한 차이는 **WE 만점 11점 → 12점**. 나머지 직접 복사.

```
mapped["WE"] = (awarded_WE / 11) × 12    # v4 → v5
mapped["그 외"] = 원본 점수 그대로
```

#### (C) v2009 / v3 → SS→LT 분리 + 4개 카테고리 비율 환산

**구조 차이**:
- LT 카테고리 **없음** (교통 크레딧이 SS 26점 안에 ~6pt 섞임)
- IP 카테고리 없음

**변환**:
```
LT      = (교통_awarded / 6)      × 15    # v5 LT max
SS      = (순수SS_awarded / 20)   × 11    # 26 - 6 = 20
WE      = (we_awarded / 10)       × 9
EA      = (ea_awarded / 35)       × 33
MR      = (mr_awarded / 14)       × 18
EQ      = (ieq_awarded / 15)      × 13
IP      = 0                               # 카테고리 없음
dropped[IN], dropped[RP]                  # v5에서 폐지, 기록만
```

#### (D) v2.2 / v2.0 → 총점 69, SS→LT 분리 + 거의 전부 비율 환산

**구조 차이**:
- 총점 69점 (v2009 이후 110점)
- LT, IP, RP 모두 **없음**
- SS 14점 중 교통 ~7pt

**변환**:
```
LT      = (교통_awarded / 7)      × 15
SS      = (순수SS_awarded / 7)    × 11    # 14 - 7 = 7
WE      = (we_awarded / 5)        × 9
EA      = (ea_awarded / 17)       × 33
MR      = (mr_awarded / 13)       × 18
EQ      = (ieq_awarded / 15)      × 13
IP      = 0
RP      = 0
dropped[IN]                               # v5에서 폐지, 기록만
```

### 4.5 SS → LT 교통 크레딧 분리 (구버전 전용)

구버전은 LT 카테고리가 없고 교통 크레딧이 SS 안에 섞여있음. v5 의 LT 로 분리하는 2단계 fallback:

**우선 1 — PDF 크레딧 상세 데이터 활용 (`credit-exact`)**

PDF 에 크레딧 상세가 파싱돼 있으면 교통 키워드 매칭으로 정확 추출.

교통 키워드:
- alternative transportation
- public transportation / transit
- bicycle
- low-emitting / fuel-efficient
- parking capacity
- green vehicles / electric vehicle

→ 해당 크레딧들의 awarded 합 = `transport_awarded`, possible 합 = `transport_max`.

**우선 2 — 역사적 평균 비율 추정 (`ratio-estimated`)**

크레딧 상세 없으면(v2009 의 114건 등) 버전별 평균 비율로 추정:

| 버전 | 교통 max (근거) |
|------|---------------|
| v2.2 | 7 (Credits 4.1=1 + 4.2=1 + 4.3=3 + 4.4=2) |
| v2009 | 6 (Credits 4.1~4.4 합계) |

```
transport_ratio   = SS_TRANSPORT_MAX / SS_total_max
transport_awarded = SS_awarded × transport_ratio
ss_pure_awarded   = SS_awarded − transport_awarded
```

⚠️ 이 추정은 **"교통 크레딧 달성률 ≈ SS 전체 달성률"** 가정에 의존. 가정이 틀릴 경우 drift 가 커져 hallucination_check FAIL → LLM 구제 경로로.

## 5. 카테고리별 "원래 만점(old_max)" 결정 우선순위

구버전 카테고리 만점(`old_max`)이 분모가 되므로 정확해야 비율 환산이 맞음. 3단계 fallback:

```
1순위: PDF의 categories_possible[cat]   (스코어카드 원본에 적힌 possible)
2순위: rubric xlsx 조회                 (data/rubrics/{version}/*.xlsx)
3순위: 버전별 BD+C 기준 하드코딩         (VERSION_BD_C_MAX)
```

### 5.1 왜 PDF possible 이 최우선인가?

**LEED 시스템(rating system) 마다 같은 카테고리도 만점이 다름**:

| 시스템 | 버전 | EA 만점 |
|--------|------|---------|
| BD+C (New Construction) | v4 | 33 |
| O+M (Existing Buildings) | v4 | **56** |
| ID+C (Commercial Interiors) | v4 | 38 |

하드코딩은 BD+C 기준만 가지고 있어서, O+M 건물을 BD+C 만점으로 나누면 달성률이 과대계산됨. PDF 스코어카드에는 **실제 시스템별 만점이 possible 필드로 명시**돼있으므로 이게 가장 신뢰할 수 있음.

### 5.2 각 단계 동작

- **1순위 (PDF possible)**: `cats_possible[cat] > 0` 이면 그 값 사용
- **2순위 (rubric xlsx)**: `data/rubrics/v4/BD+C_NewConstruction.xlsx` 같은 공식 스코어카드에서 조회. 모듈 로드 시 `data/rubrics/` 아래 모든 xlsx 스캔·캐시. 파일 없으면 자동 skip.
- **3순위 (하드코딩)**: BD+C 기준 기본값. 루브릭 파일 없는 버전/시스템에서도 에러 없이 동작 보장.

## 6. `mapping_rules.yaml` — 크레딧 단위 매핑

### 6.1 역할

4절의 비율 환산은 **카테고리 합계** 레벨 변환. 그러나 "v2.2 의 어떤 크레딧이 v5 의 어떤 크레딧에 대응되는가" 같은 **크레딧 단위 매핑**은 수식이 아니라 규칙 테이블로 해결. 이게 `mapping_rules.yaml`.

### 6.2 파일 구조

현재 **107개 규칙**. 카테고리별 섹션 (EA / WE / SS / LT / MR / EQ / IP / O+M 전용 / 폐지).

각 규칙 7개 필드:

```yaml
- source_credit: "optimize energy performance"   # 구버전 크레딧명 (소문자 부분 매칭)
  source_versions: [v2.2, v2009, v3, v4, v4.1]  # 이 규칙이 적용되는 버전
  source_category: EA                            # 구버전 카테고리
  target_v5_code: EAc3                           # v5 크레딧 코드
  target_v5_name: Enhanced Energy Efficiency     # v5 크레딧명
  target_v5_category: EA                         # v5 카테고리
  confidence: high                               # high / medium / low
```

### 6.3 매칭 알고리즘

```
입력: PDF 크레딧명 "Credit: Optimize Energy Performance 18pt"
      원본 버전 "v4"

1. 전처리: 소문자화, "Credit:" 등 접두사 제거
   → "optimize energy performance 18pt"

2. YAML 의 모든 source_credit 과 부분 문자열 매칭 검사
   - "optimize energy performance" ⊂ "optimize energy performance 18pt" ✓

3. source_versions 에 현재 버전 포함된 것만 필터
   - [v2.2, v2009, v3, v4, v4.1] 에 "v4" 포함 ✓

4. 여러 규칙 매칭 시 source_credit 이 가장 긴 것 우선 (specificity)
   - "energy performance" (18자) vs "optimize energy performance" (27자)
     → 후자 채택

5. 결과: EAc3 / Enhanced Energy Efficiency / EA
```

### 6.4 출력 — 크레딧 매핑 결과

각 크레딧마다 다음 정보가 기록돼 `standardized_credits.parquet` 로 저장:

| 컬럼 | 예시 |
|------|------|
| credit_name | Credit: Optimize Energy Performance 18pt |
| awarded / possible | 15 / 18 |
| v5_code | EAc3 (매칭 실패 시 UNKNOWN) |
| v5_name | Enhanced Energy Efficiency |
| v5_category | EA |
| confidence | high / medium / low |
| matched | True / False |

### 6.5 매핑 성공률 (`credit_rule_hit_rate`)

각 프로젝트마다 `hits / (hits + misses)` 로 커버리지 측정.

| 버전 | 평균 hit rate (참고) |
|------|---------------------|
| v4 | 88% |
| v4.1 | 85% |
| v2009 | 64% |
| v2.2 | 70% |

낮은 프로젝트는 분석 robustness 체크 시 `credit_hit > 0.7` subset 으로 따로 검증 (02 문서 §11).

## 7. 수학 검증 (Hallucination Checker)

### 7.1 왜 "hallucination check" 인가?

LLM 환각 검사가 아니라 **Rule 매핑 결과의 수학 제약 검사**. LLM 안 씀.

### 7.2 5가지 검증 항목

| # | 검증 항목 | 예시 FAIL |
|---|----------|----------|
| 1 | 카테고리 점수 ≤ v5 최대값 | EA = 35 (v5 EA max = 33) → FAIL |
| 2 | 총점 일관성 (sum ≈ total, 오차 0.5 이내) | 합=61, reported=65 → FAIL |
| 3 | **달성률 드리프트 ≤ 20%** | 원본 60% → v5 85% (+25%) → FAIL |
| 4 | 음수 점수 없음 | LT = -2 → FAIL |
| 5 | v5 에 없는 카테고리 없음 | mapped 에 IN 남아있음 → FAIL |

5개 전부 통과해야 PASS. 하나라도 실패 시 LLM 구제 경로.

### 7.3 드리프트(drift) 란 — 핵심 지표

**정의**:
```
원본 달성률 = total_score_raw / version_total_max
v5 달성률   = total_score_v5  / v5_total_max
drift      = |원본 달성률 - v5 달성률|
```

**예시**:
- 원본(v2.2): 55점 / 69점 = **79.7%**
- 환산(v5): 75점 / 100점 = **75.0%**
- drift = |79.7% - 75.0%| = **4.7%** → 20% 이내, PASS

**의미**: "버전 환산이 이 건물의 실제 실력을 얼마나 왜곡했는가".
- drift = 0% → 완벽 보존
- drift 증가 → 환산 과정에서 점수 과대/과소 평가 발생
- drift > 20% → 신뢰 불가, LLM 구제 필요

### 7.4 왜 20% 임계치인가?

1. **LEED 등급 간격과 비교**: Certified(40~49) vs Silver(50~59) 간격 10점 = 달성률 10%. 20% drift 는 **등급 2단계 이동** 가능 = 치명적 왜곡.
2. **v2009 크레딧 상세 부재 114건**: 카테고리 비례배분으로 추정 → 구조적 drift 불가피. 10% 엄격 기준이면 전부 탈락.
3. **실증 tuning**: 처음엔 10%로 시작했으나 LLM 호출 비용 폭증 → 20% 로 완화.

### 7.5 drift 가 커지는 실제 원인

1. **SS→LT 교통 분리 비율 추정 오류** (가장 흔함): 크레딧 상세 없는 v2009 에서 교통 비율을 잘못 추정
2. **PDF 파싱 오류**: possible 값 추출 실패 → old_max 오판
3. **시스템 오매칭**: O+M 건물을 BD+C 만점으로 분모 계산
4. **버전 미식별**: `version = "unknown"` → v4 fallback 으로 잘못된 분모

### 8.2 검증경로 분기 (7.0%) 가 나오는 로직

460개 PDF 각각:
```
pdf_ingest → csv_match → rule_mapper → hallucination_checker
                                            ↓
                                   math PASS → rule 점수 (track="rule")
                                   math FAIL → LLM Mapper (track="llm")
```

`hallucination_checker` 의 drift ≤ 20%p 제약 기준:
- math PASS (drift ≤ 20%p) = **428건 (93.0%)** → Track 1
- math FAIL (drift > 20%p) = **32건 (7.0%)** → Track 2

단, 커밋된 `project_features.parquet` 는 OPENAI_API_KEY 없이 rule-only 로
재생성되어 `standardization_track` 이 460건 모두 `"rule"` 이다 (graceful
degradation). 위 7.0% 는 API 사용 시 LLM 검증경로로 분기되는 표본 비율.

### 8.3 검증경로 분기 32건의 특징

- IP 파싱 수정 후 drift > 20%p 표본은 **거의 전부 v4** (31건) + v2.2 1건
- v4 고급 소매(ID+C Retail) 등에서 설비 투자는 큼(EA 높음)인데 SS/LT 제약이
  낮아 원본↔v5 달성률 격차(drift)가 크게 벌어지는 패턴
- drift 최대 48%p, 20%p 임계 초과가 분기 조건

## 9. 최종 출력 — `project_features.parquet`

### 9.1 주요 컬럼 그룹 (총 28 컬럼)

**식별**
- `project_id`, `project_name`, `leed_system`, `building_type`, `gross_area_sqm`

**원본**
- `original_version`, `certification_level`, `total_score_original`
- `achievement_ratio_original`

**v5 매핑**
- `total_score_v5`, `achievement_ratio_v5`

**ML feature — 카테고리별 달성률 (0~1)**
- `ratio_LT`, `ratio_SS`, `ratio_WE`, `ratio_EA`, `ratio_MR`, `ratio_EQ`, `ratio_IP`

**카테고리 절대점수 (비교용, ML 에서 미사용)**
- `score_v5_{cat}`, `score_orig_{cat}`

**메타**
- `standardization_track` — `"rule"` 또는 `"llm"`
- `standardization_iterations`
- `mapping_rationale`

**LLM 리뷰 메타 (Option A — 점수는 바꾸지 않음, 분석용 signal)**
- `llm_review_target` — `"rule"` 또는 `"llm"`
- `llm_review_is_valid`
- `llm_review_score` — 0~1
- `llm_review_issues`
- `llm_review_feedback`

### 9.2 `ratio_{cat}` 이 핵심 ML feature 인 이유

```
ratio_{cat} = score_v5_{cat} / v5_max[cat]
            = (awarded / old_max × v5_max) / v5_max
            = awarded / old_max
            = "이 카테고리에서 원본 기준 몇 % 달성했나"
```

**버전이 달라도 동일 의미** (달성률) → ML 모델에 직접 투입 가능.
