> LangGraph 기반 이중 검증 파이프라인, RandomForest + SHAP 로 등급 결정 요인 분석.

# Part 1. LangGraph 파이프라인 설계

## 1. 왜 LangGraph 인가?

각 단계를 **독립 노드로 분리** + **조건부 엣지로 분기 정책 집중** + **State 로 중간 결과 추적**.

| 방식 | 장점 | 단점 |
|------|------|------|
| 단순 스크립트 | 간단 | 중간 결과 추적 어려움, 분기 조건 산만 |
| Rule 만 | 재현성 100% | 의미 검증 불가능 |
| LLM 만 | 유연함 | 매번 결과 다름, 비용 높음, 책임소재 불명 |
| **LangGraph + Rule + LLM (이 연구)** | **재현성 + 의미 검증 동시** | 구현 복잡 |

---

## 2. 파이프라인 구조 (노드 7개)

### 2.1 노드 목록

| # | 노드 | LLM | 역할 |
|---|------|-----|------|
| 1 | `pdf_ingest` | No  | PDF 파싱 (pdfplumber + 정규식) |
| 2 | `csv_match` | No  | 프로젝트 디렉토리 CSV 조인 |
| 3 | `rule_mapper` | No  | 결정론적 v5 매핑 (비율 환산) |
| 4 | `hallucination_checker` | No  | 수학 검증 (5개 제약) |
| 5 | `llm_validator` | Yes | LLM 의미 검증 (target=rule / target=llm) |
| 6 | `llm_mapper` | Yes | LLM 재매핑 (math FAIL 구제 경로) |
| 7 | `finalize` | No  | 최종 저장 |

### 2.2 전체 흐름도

```
  [Track 1: 결정론적 - 93.0%]
  pdf_ingest → csv_match → rule_mapper → hallucination_checker
                                              ↓ math PASS
                                         llm_validator(target=rule)
                                              ↓
                                           finalize
                                  * LLM 은 검토만, 점수는 rule 결과 유지

  [Track 2: LLM 구제 - 7.0%]
  ... rule_mapper → hallucination_checker
                           ↓ math FAIL
                       llm_mapper → llm_validator(target=llm)
                                         ↓
                                      finalize
                          * 이 경로에서만 LLM 점수를 최종 채택

  [Graceful degradation]
  OPENAI_API_KEY 없음 → 무조건 finalize (rule 결과만 저장)
```

## 3. 각 노드 상세

### 3.1 `pdf_ingest` — PDF 안의 숫자 추출

- 입력: PDF 경로
- 처리: `pdfplumber` 로 전체 텍스트 추출 → 정규식으로 project_id / version / categories / credits 파싱
- 출력: 파싱된 dict (project 정보, 카테고리별 awarded/possible, 크레딧 목록)

상세는 [01_전처리_과정.md §3](01_전처리_과정.md) 참조.

### 3.2 `csv_match` — 프로젝트 메타 보강

- PDF 에서 추출한 project_id / project_name 으로 디렉토리 CSV 조인
- 매칭 성공 시: 인증 등급, 연면적(sq ft → sqm 변환), 건물유형 등 보강
- 매칭 실패 시: PDF 데이터만으로 project 구성 (분석에서 제외하지 않음)

### 3.3 `rule_mapper` — 결정론적 v5 매핑

**핵심 공식**:
```
score_v5(cat) = (awarded / old_max) × v5_max(cat)    # v5_max 로 클램핑
```

**버전별 분기**:
- v4.1 / v5 → 직접 매핑
- v4 → WE 만 11→12 비율 환산
- v2009 / v3 → SS→LT 분리 + WE/EA/MR/IEQ 비율 환산 + IP = 0
- v2.2 / v2.0 → SS→LT 분리 + 거의 전부 비율 환산

**SS→LT 교통 분리**: PDF 크레딧 상세 있으면 키워드 매칭 (`credit-exact`), 없으면 역사적 비율 추정 (`ratio-estimated`: v2.2=7pt, v2009=6pt).

**크레딧 단위 매핑**: `mapping_rules.yaml` 의 107개 규칙으로 개별 크레딧을 v5 크레딧 코드에 매핑. `credit_rule_hit_rate` 로 커버리지 측정.

상세는 [01_전처리_과정.md §4~6](01_전처리_과정.md) 참조.

### 3.4 `hallucination_checker` — 수학 검증

**"hallucination check" = LLM 환각 검사가 아니라 Rule 매핑 결과의 수학 제약 검사.** LLM 안 씀.

**5가지 검증**:
1. 카테고리 점수 ≤ v5 최대값
2. 총점 일관성 (sum ≈ total, 오차 0.5 이내)
3. 달성률 **drift** ≤ 20%
4. 음수 점수 없음
5. v5 에 없는 카테고리 없음

**드리프트(drift)**:
```
drift = |원본 달성률 − v5 달성률|

원본 달성률 = total_score_raw / version_total_max
v5 달성률   = total_score_v5  / v5_total_max
```

drift 가 20% 초과 시 환산이 건물 실력을 왜곡했다는 신호. 20% 임계치 근거는 [01_전처리_과정.md §7.4](01_전처리_과정.md) 참조.

**라우팅**:
```
math PASS + OPENAI_API_KEY 있음 → llm_validator(target=rule)
math FAIL + OPENAI_API_KEY 있음 → llm_mapper (구제 경로)
OPENAI_API_KEY 없음            → 무조건 finalize (graceful degradation)
```

### 3.5 `llm_validator` — LLM 의미 검증 (두 모드)

**역할**: 점수를 바꾸지 않음. **LLM 은 심사자이지 계산자가 아님**. 두 가지 모드로 분기 (프롬프트 자체가 다름):

#### target = "rule" — Rule 결과의 의미적 타당성 검토

입력은 수학적으로 이미 통과된 Rule 매핑. LLM 은 **도메인 전문가 관점**에서 검토:

1. LEED 버전 특성 반영 (v2.2 SS→LT 분리가 합리적?)
2. 크레딧 체계적 누락 여부
3. v5 신규 IP(Integrative Process) 배분이 건물 특성에 적절?
4. 원본 등급·건물 유형·연면적과 환산 점수 정합성
5. 원본에 없는 카테고리(v2.2→LT/RP/IP)에 점수 잘못 부여 여부

> **"Rule 이 수학적으로 맞아도 의미가 틀리면 is_valid=false"** 명시.

#### target = "llm" — LLM 재매핑 결과의 할루시네이션·수치 검증

math FAIL 로 `llm_mapper` 가 재매핑한 결과 검증. **LLM 이 또 헛소리 안 했는지** 체크:

1. 카테고리 ≤ v5 최대값
2. 인증 등급 일관성 (Certified 40~49, Silver 50~59, Gold 60~79, Platinum 80+)
3. 달성률 drift ≤ 20%
4. 원본 버전에 없는 카테고리에 점수 부여 금지

#### 두 target 차이 요약

| target | 검증 대상 | 초점 | 배경 |
|--------|----------|------|------|
| **rule** | Rule 매핑 결과 | 의미적 타당성 | math 는 이미 통과 — "숫자가 맞아도 LEED 맥락에서 말 되는가?" |
| **llm** | LLM 재매핑 결과 | 수치·할루시네이션 | LLM 출력이라 수식 제약부터 의심 |

### 3.6 `llm_mapper` — LLM 재매핑 (구제 경로)

**진입 조건**: `hallucination_checker` math FAIL (rule 계산이 수학적으로 문제 있음).

**처리**:
- Rule 결과, math 이슈, 이전 LLM 피드백을 모두 context 로 전달
- 버전별 매핑 가이드 프롬프트 포함
- JSON 강제 응답 → 카테고리별 점수 + 근거
- LLM 출력도 v5_max 로 클램핑 (LLM 환각 2차 방지)
- rate limit 대응: exponential backoff 최대 5회 (2→32초) + 호출 간 2초 sleep
- 재시도 실패 시 rule 결과로 fallback

### 3.7 `finalize` — 최종 저장

- `validation_target` 기준으로 최종 결과 선택:
  - `"rule"` → Rule 매핑 결과 채택 (Track 1, 93.0%)
  - `"llm"` → LLM 재매핑 결과 채택 (Track 2, 7.0%)
- 달성률·LLM 리뷰 메타 포함해 `project_features.parquet` 에 1행 추가

## 4. Option A — "LLM 은 점수를 바꾸지 않는다" 의 정확한 의미

### 4.1 원칙

> **Track 1 (math PASS, 93.0%) 에서는 LLM 이 Rule 결과를 거부해도 Rule 점수가 유지된다.**

LLM validator 판정은 **finalize 라우팅에 영향을 주지 않음**. 무조건 finalize 로 이동.

LLM 판단은 **메타데이터로만 기록**:
- `llm_review_target` — `"rule"` 또는 `"llm"`
- `llm_review_is_valid` — LLM 의 판단
- `llm_review_score` — 신뢰도 0~1
- `llm_review_issues` — 지적 사항
- `llm_review_feedback` — 전체 의견

→ 점수엔 반영 안 되지만 분석용 **uncertainty signal** 로 활용 가능 (LLM 이 거부한 건만 뽑아 subset 분석 등).

### 4.2 예외 — Track 2 (math FAIL, 7.0%) 에서는 LLM 재매핑 결과 사용

> **Rule 이 수학 제약조차 통과 못 하면 (drift > 20% 등) Rule 점수 자체를 신뢰할 수 없음 → 이때만 LLM 재매핑 결과를 최종 채택.**

이게 바로 **"LLM 구제 경로"**. 460 건 중 32 건(7.0%) 이 이 경로를 탄다 (IP 파싱 수정 후 drift>20%p 표본 기준).

### 4.3 이전 대화 혼동 정리

- **Track 1** (93.0%): LLM 은 검토만, Rule 점수 유지 ← **"LLM 은 점수를 안 바꾼다"** 는 여기 적용
- **Track 2** (7.0%): Rule 자체가 실패해서 LLM 이 재매핑 → LLM 점수 채택

즉 **"LLM 재매핑"은 math FAIL 된 7.0% 에 대해서만 발생**. 메인 경로(93.0%)에서는 LLM 이 점수에 영향 X.

### 4.4 왜 Option A 인가?

1. **재현성**: 점수는 항상 Rule (93.0%) → 동일 입력 = 동일 출력
2. **논문 본론 유지**: "LLM 매핑 정확도" 증명 부담 회피 (NLP 논문화 방지)
3. **책임소재 명확**: 계산 = Rule, 검증 = LLM → 수치 출처 추적 가능
4. **비용**: LLM 주계산자면 수백 달러. 검증만이면 훨씬 저렴

---

## 5. Rate Limit 대응

- Exponential backoff 최대 5회 (2→4→8→16→32초)
- 호출 간 `sleep(2)` (OpenAI TPM 30k 기준)
- 최종 실패 시 → `llm_mapper` 는 rule 결과로 fallback, `llm_validator` 는 강제 승인
- `max_iterations` 도달 시 강제 승인 (무한 루프 방지)

## 8. 검증

**5-Fold Stratified CV** (등급 분포 보존). Train/Test split 대신 CV 쓴 이유:
- N=460 이 작아 단일 split 분산 큼
- 등급 분포 편향 방지 (Certified 11%)

## 9. 성능

| 지표 | 값 |
|------|-----|
| **CV Accuracy** | **0.8457 ± 0.0081** |
| **CV Weighted F1** | **0.8401 ± 0.0089** |
| Train Accuracy | 0.998 (과적합 — 논문 미사용) |

Train 0.998 / CV 0.846 으로 gap 약 15%. 그래서 **논문엔 CV 값만 사용**. Holdout test set 미도입은 **후속 과제**로 명시.

# Part 3. 심사 예상 Q&A

### Q1. 왜 LangGraph? 함수로 해도 되잖아요

각 단계가 **독립 노드로 분리**돼 디버깅 쉬움. 조건부 엣지로 분기 정책 한곳에 집중. State 로 중간 결과 전부 추적. 체크포인트로 장시간 실행 중 끊겨도 resume 가능.

### Q2. 왜 Rule 먼저, LLM 나중?

1. **재현성**: Rule 은 동일 입력 = 동일 결과. LLM 은 매번 다름.
2. **비용**: LLM 주계산자면 수백 달러. Rule 은 API 0회.
3. **책임소재**: 계산 = Rule, 검증 = LLM 분리 → 수치 출처 명확.

### Q3. LLM 이 Rule 을 90% 거부했는데 Rule 이 틀린 거 아니냐?

- threshold 0.8 이 보수적이라 거부율 높음 (75건 실험 기준)
- LLM 승인 7건은 **구체적 도메인 근거** 제시 ("LT/SS 분리 올바름, IN/RP 폐지 정확히 반영")
- Option A 채택으로 **점수는 항상 Rule**, LLM 은 uncertainty signal 만 제공

### Q4. 460개로 일반화 되나?

USGBC Public LEED Project Directory 기준 **한국 전수**. 표본 추출이 아니라 한국 시장 대표. 해외 확장 시 LLM 프롬프트의 국가 맥락만 교체.

### Q5. Train 정확도 1.0 은 과적합 아닌가?

맞음. Train 0.998 / CV 0.8457 로 gap 약 15%. 그래서 논문엔 CV 값만 사용. Holdout test set 미도입 — **후속 과제**로 명시.

### Q6. v2009 114 건은 왜 크레딧 분석에서 빠졌나?

v2009 PDF 가 **카테고리 합계만 제공**, 크레딧 상세 없음. 카테고리 레벨 분석엔 포함되지만 크레딧 레벨 SHAP 에서는 제외.

### Q7. SS→LT 분리 비율(v2.2=7, v2009=6) 의 출처는?

- v2.2 SS Credits 4.1(1) + 4.2(1) + 4.3(3) + 4.4(2) = 7점
- v2009 SS Credits 4.1~4.4 = 6점

USGBC LEED Reference Guide 각 버전 SS 챕터 근거. 하드코딩된 값이라 **Peer review 필요** — 한계로 명시.

### Q8. drift 20% 가 왜 기준?

1. LEED 등급 간격(Certified/Silver/Gold/Platinum) 폭 = 10점 = 10%. 20% drift = 등급 2단계 이동 가능.
2. v2009 크레딧 상세 없는 114건을 처리하려면 10% 로는 너무 엄격.
3. 10% 로 시작했으나 LLM 호출 비용 폭증 → 20% 로 실증 tuning.

### Q9. 매핑 규칙 107개는 누가 검증?

LEED 공식 루브릭 기반 연구자 수동 작성. **Peer review 미실시**. `credit_rule_hit_rate` 로 커버리지 정량화 (v4 88%, v2009 64%).

### Q10. CV fold 분산은?

RandomForest 기준 fold 별 83.7~85.9% (표준편차 0.008) 로 매우 안정적.
트리 앙상블 평균화 효과로 이전 XGBoost(표준편차 0.045) 대비 fold 분산이
크게 줄었다. 다만 N=460 이 feature space 대비 작은 점은 여전한 한계.

후속: 층화 기준에 버전 추가.

### Q11. Robustness 의 'LLM 재매핑(drift>20%)' 32건은?

rule 매핑이 `hallucination_checker` 의 drift ≤ 20%p 제약을 통과하지 못해
검증경로(LLM 재매핑) 대상이 되는 표본군 (460건 중 32건, 7.0%). 표본이
작아 3-Fold CV 를 적용했으며, SHAP 1위가 log_area 로 나오나 표본 부족에
따른 잡음으로 본다. 후속: drift 임계치 tuning + 검증경로 표본 확대.

### Q12. 이 연구의 기여?

1. **버전 전수** (v2.0~v4.1)
2. **LangGraph 이중 검증** 재현 가능 파이프라인
3. **한국 전수 460건** 공개 데이터셋
4. **SHAP 크레딧 단위** 상세 분석

### Q13. 후속 연구 방향?

1. Holdout test set + 외부 검증
2. LLM 전수 리뷰 (threshold 완화)
3. 추가 feature (용도, 위치, 준공 연도)
4. 해외 데이터셋 확장
5. 구버전 매핑 정교화 (SS→LT 비율 peer review)
